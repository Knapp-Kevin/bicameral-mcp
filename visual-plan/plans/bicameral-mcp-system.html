00 / Motivation

Five SDLC Friction Points

Every software team makes hundreds of verbal decisions per week. Those decisions live in meeting transcripts, PRDs, Slack threads, and people's heads — none of which are linked to the code that implements (or fails to implement) them. General-purpose AI can shred a PRD into user stories, but it goes deaf the moment code hits production. Bicameral's wedge is post-commit drift detection — knowing that a decision made three weeks ago is now inconsistent with what actually shipped.
CONSTRAINT_LOST
Known limits surface mid-sprint
A rate limit, auth model constraint, or compliance rule was discussed — but only discovered by engineering after implementation is underway. Result: rework, scope creep.
Solved by: bicameral.search → pre-flight context before coding
CONTEXT_SCATTERED
The “why” is split across 4 tools
An architectural decision's rationale lives in a Slack thread, a huddle recording, someone's memory, and a Notion doc. No single source of truth. Wrong calls when that person leaves.
Solved by: bicameral.ingest → normalized intent graph from any source
DECISION_UNDOCUMENTED
Verbal “let's do X” never lands in a ticket
“We're going collaborative not adversarial” is said in a meeting but never written down. Implementation diverges from intent because intent was never captured.
Solved by: bicameral.ingest + bicameral.status → track what was decided vs built
REPEATED_EXPLANATION
Same context tax paid twice
PM explains intent to design, then re-explains to engineering. Engineer re-discovers constraints PM already surfaced. Handoff latency, PM frustration.
Solved by: bicameral.search + get_neighbors → retrieve full decision provenance
TRIBAL_KNOWLEDGE
Only one person knows why
The system works the way it does because of a conversation that happened 6 months ago. That person is on vacation. Key-person risk, onboarding drag.
Solved by: bicameral.drift + bicameral.status → institutional memory in code

How the Tools Recombine

Each tool is a composable primitive. The host model (Claude, Cursor) chains them into workflows that address the failure modes above. Three canonical flows:
Pre-flight (Planning)
Before writing code
bicameral.search(feature description)
→ surfaces past decisions + constraints

validate_symbols(hypothesis)
→ confirms code entities exist

search_code(query)
→ locates relevant code regions

get_neighbors(symbol_id)
→ understands blast radius
Fixes: CONSTRAINT_LOST, REPEATED_EXPLANATION
Code Review (Quality)
Before merging code
bicameral.drift(changed_file)
→ surfaces decisions that touch this file

extract_symbols(file_path)
→ enumerates what's in the file now

bicameral.status(filter=“drifted”)
→ full drift report
Fixes: DECISION_UNDOCUMENTED, TRIBAL_KNOWLEDGE
Ingestion (Memory)
After a meeting or sync
bicameral.ingest(transcript_payload)
→ extracts intents, maps to code

bicameral.link_commit(HEAD)
→ syncs latest commit state

bicameral.status(since=meeting_date)
→ shows what's reflected vs pending
Fixes: CONTEXT_SCATTERED, DECISION_UNDOCUMENTED
01 / System Overview

Architecture at a Glance

The MCP server is split into two subsystems. The Decision Ledger (5 tools) manages a SurrealDB knowledge graph of intent → symbol → code_region with content-hash drift detection. The Code Locator (4 tools) provides deterministic code search over a SQLite-backed index with BM25, graph traversal, and tree-sitter extraction.
HOST MODEL (Claude, Cursor, etc.) | | MCP stdio transport | v server.py — MCP entrypoint, tool dispatch | |——————————————————————————————————————| | | DECISION LEDGER (5 tools) CODE LOCATOR (4 tools) | | handlers/ adapters/code_locator.py decision_status.py RealCodeLocatorAdapter search_decisions.py | detect_drift.py code_locator/ link_commit.py tools/ — validate, search, neighbors ingest.py indexing/ — symbol_extractor, sqlite_store | retrieval/ — bm25s, sqlite_vec adapters/ledger.py fusion/ — RRF SurrealDBLedgerAdapter (singleton) | | SQLite (symbols, BM25 index, graph edges) ledger/ adapter.py, client.py, queries.py schema.py, status.py | SurrealDB v2 (embedded via Python SDK) surrealkv:// (persistent) or memory:// (tests)
No nested LLM calls. The host model calls MCP tools directly and performs reasoning itself.

04 / Code Locator

Deterministic Code Search

The code locator indexes a repository into a SQLite database with three retrieval channels: BM25 text search (bm25s), structural graph traversal (call/import/inherit edges from tree-sitter), and optional vector similarity (sqlite-vec + sentence-transformers). Results are fused via Reciprocal Rank Fusion (RRF).
INDEXING PIPELINE (runs on HEAD mismatch or first use) source files → tree-sitter → SymbolDB (SQLite: symbols, files, edges) → BM25 index (bm25s serialized to disk) → vector index (sqlite-vec, optional) Guard: index_meta.head_commit tracks the last indexed HEAD. On mismatch: full re-index (incremental planned via CocoIndex). RETRIEVAL (per search_code call) query → BM25 channel → ranked file:line hits query → graph channel → symbol_ids → 1-hop expansion → code_regions query → vector channel → embedding similarity (when available) | v RRF fusion → top-k results with provenance ANTI-HALLUCINATION Every returned file path is verified against the indexed state. Every symbol_id maps to a real row in SymbolDB. No LLM generates file paths or symbol names.
Index
SymbolDB (SQLite)
Stores all extracted symbols with qualified names, file paths, line ranges, and type (function/class/module). Graph edges stored as (src_id, dst_id, edge_type) triples.
code_locator/indexing/sqlite_store.py
Search
BM25 (bm25s)
Full-text search over symbol names and file content. Tokenized and serialized to disk. Fast sub-50ms retrieval.
code_locator/retrieval/bm25s_client.py
Fusion
RRF (Reciprocal Rank Fusion)
Combines ranked lists from BM25, graph, and vector channels. k=60 default. Produces a single ranked list with per-result provenance tags.
code_locator/fusion/rrf.py
Extraction
tree-sitter
Parses 7 languages (Python, JS, TS, Java, Go, Rust, C#). Extracts function/class/module symbols with line ranges. Builds call/import/inherit graph edges.
code_locator/indexing/symbol_extractor.py
05 / Decision Ledger

SurrealDB Knowledge Graph

The ledger maintains a parallel history to git. Bicameral versions intent; git versions code. They are connected by a single primitive: sha256(git show <ref>:<file>[start:end]). Status is never stored — it is always re-derived from a hash comparison at query time. This makes Bicameral immune to rebase, squash, and cherry-pick.
// The two timelines SPEC TIMELINE (Bicameral — append-only) CODE TIMELINE (Git — branching DAG) intent:A "webhook retry backoff" main: c1 —— c2 —— c3 —— M1 —— c7 intent:B "rate limit checkout" \ / intent:C "optimistic lock cart" feat: c4 —— c5 —— c6 CONTENT HASH BRIDGE sha256(git show <ref>:<file>[start_line:end_line]) status(intent, ref) = no code_region → "ungrounded" actual_hash=None → "pending" (symbol absent at ref) actual ≠ stored → "drifted" (code changed since baseline) actual = stored → "reflected" (code matches intent)
Status is a pure function of (intent, git_ref) — never stored, always re-derivable.

06 / Knowledge Graph

The Graph Shape

Three node types and three edge types encode the full provenance chain from meeting decision to code region.
// Example: one meeting, three decisions intent:A [status:"drifted"] "webhook retry with exponential backoff" | maps_to (conf:0.91) v symbol:WebhookDispatcher.send | implements (conf:0.88) v code_region:A [webhook/dispatcher.py:134-180, hash:"a3f2..."] | depends_on (invokes) v code_region:B [payments/processor.py:45-92] ← shared dependency ^ | implements ← symbol:PaymentProcessor.process ^ intent:B [status:"pending"] | maps_to (conf:0.85) "rate limit checkout endpoint" ─────────────────────┼
code_region:B is a shared structural dependency — blast radius node for intents A and B.
Node
intent
A decision from a meeting or PRD. Fields: description, feature_hint, source_ref, meeting_date. BM25 index on description.
SEARCH ANALYZER biz_analyzer BM25(1.2, 0.75)
Node
symbol
A named code entity (function, class). The vocabulary bridge between intent and code. BM25 index on name (CamelCase + snake_case aware).
SEARCH ANALYZER code_analyzer BM25(1.2, 0.75)
Node
code_region
A specific span (file, start_line, end_line) tied to a symbol. Carries content_hash for drift detection.
content_hash = sha256(lines[start:end])
Edge
maps_to
intent → symbol. Created by ingest_payload. Carries confidence from code-locator bridge matching.
RELATE intent:A->maps_to->symbol:B
Edge
implements
symbol → code_region. The join between vocabulary space and code space.
RELATE symbol:B->implements->code_region:C
Edge
depends_on
code_region → code_region. Structural dependency from tree-sitter AST. Enables blast radius queries.
edge_type: invokes | imports | inherits
07 / Status Derivation

Four States, Zero Stored Flags

Decision status is a pure function computed at query time. No status field is ever written to the database.
Reflected
Code matches intent
content_hash at current ref equals the stored hash. The decision is fully implemented and unchanged.
Drifted
Code diverged from intent
content_hash differs. The code region has been modified since the decision was recorded. Needs review.
Pending
Symbol absent at ref
The mapped code region does not exist at the current git ref. Implementation not yet started or was deleted.
Ungrounded
No code region mapped
The intent has no maps_to → symbol → code_region chain. It was ingested but the code locator found no matching code.
