Git Truth, Ledger Truth,
And What Ships Today

This plan uses docs/architecture/version-control-integration.md as the design reference, but it is grounded in the code that exists now in pilot/mcp. The core shipping model is: committed state is git-addressed, working-tree changes are ephemeral overlays, and the local code index auto-refreshes when its recorded HEAD no longer matches the repo.

1. Productized Command Surface

The internal MCP tool IDs still use the current names, but the productized user-facing command namespace should be slash-prefixed and product-scoped. This is the command language the packaged MCP should present.

/bicameral:statusShow decision implementation status for the current repo baseline.
/bicameral:searchSurface prior decisions and grounded code regions before implementation.
/bicameral:driftCheck a file against tracked decisions in the working tree or at HEAD.
/bicameral:link-commitSync one git commit into the ledger and update affected statuses.
/bicameral:ingestPlanned future command for transcript/product-discussion ingest.

2. Operating Model

Committed baseline
main

Production-facing answers anchor on committed git state. link_commit and status derivation treat git refs as the durable truth.

Ephemeral overlay
working tree

Local modifications matter for pre-commit drift review, but they do not redefine the canonical committed baseline.

Repo-local state

$REPO/.bicameral/ owns the SQLite code index, BM25 index, and runtime caches. That keeps third-party installs repo-scoped rather than home-directory global.

Git-aware code index

pilot/mcp/code_locator_runtime.py persists repo_path, head_commit, and branch into index_meta.

Content-hash ledger

pilot/mcp/ledger/status.py derives reflected, drifted, pending, and ungrounded from hash comparison rather than branch-specific stored status.

5. What Happens On Common Git Events

Checkout / merge / pull

  1. The next real Code Locator query reads index_meta.head_commit.
  2. If the recorded commit differs from current git rev-parse HEAD, MCP forces an index rebuild.
  3. BM25 is rebuilt alongside SQLite metadata so stale symbol/path answers do not leak through.

link_commit

  1. The ledger resolves HEAD to an actual SHA.
  2. git show <sha> --name-only identifies changed files.
  3. For tracked regions in those files, it recomputes hashes and updates intent status.

detect_drift (working tree)

  1. The tool auto-syncs committed baseline with link_commit(HEAD).
  2. It traverses the ledger for decisions touching the target file.
  3. It extracts current file symbols from disk to show undocumented local work.

decision_status

  1. Reads current ledger graph state.
  2. Does not auto-sync.
  3. Reports the status already derived and stored by ingest/link flows.

6. Implemented vs Not Yet Implemented

Cross-branch compare without checkout Not implemented No detached git-ref indexing yet. MCP still indexes the checked-out working tree snapshot.
Blob-addressed dedupe across branches Not implemented No branch-agnostic embedding cache yet.
MCP-native retrieval surface with no nested LLM Planned This is the key remaining productization step for model/runtime boundaries.

7. Release Readiness For VC Behavior

Ready now

Repo-local state, stale-index auto-refresh, real commit sync, and end-to-end tests with real adapters are all in place.

Safe default for downloads

Third-party installs should default to $REPO/.bicameral and document REPO_PATH, CODE_LOCATOR_SQLITE_DB, and provider credentials explicitly.

Still a product gap

The packaged MCP is not fully modular until the deprecated code-locator runtime is pulled into pilot/mcp and the nested LLM path is removed.
