Code Locator System Overview

Selection-based code retrieval for Bicameral. Translates natural-language planned changes from meeting transcripts into grounded code locations with full provenance. The LLM never invents file paths or symbols — it proposes candidates that are validated against a real index.

01 / Core Principle

Selection, Not Generation

The LLM reasons about what to search for, but every code fact (file path, symbol name) comes from tool results backed by a real index. The final answer is validated against an allowlist — anything not seen in tool results is rejected. Zero hallucinated paths reach the output by construction.

02 / Indexing Pipeline

Two Backends, One Schema

Feature-flagged via indexing_backend. The CocoIndex backend (Option A+) runs a single pipeline that produces both symbols and embeddings with fingerprint-based incrementality. Everything lives in one SQLite file. As of 2026-03-28, that SQLite file also records the git state it was built from so query-time usage can auto-refresh on HEAD mismatch.
// Option A+ (indexing_backend="cocoindex") — RECOMMENDED CocoIndex v1 Pipeline (cocoindex_pipeline.py) walk_dir(repo, recursive=True, path_matcher=...) | +-- @coco.fn(memo=True) process_file(file) ← skips unchanged files | | | +-- RecursiveSplitter.split(content, chunk_size=512) | | +-- SentenceTransformerEmbedder.embed(chunk) → float[384] | | +-- code_embeddings_vec (sqlite-vec / vec0 virtual table) | | | +-- extract_symbols_from_content(content, language, path) | +-- tree-sitter parse → cocoindex_symbols (regular table) | +-- sync_symbols_in_db() → INSERT SELECT (same DB, microseconds) | +-- symbols + indexed_files (legacy SymbolDB tables) | +-- graph_builder.build_graph() → 2nd tree-sitter pass | +-- contains edges (class → method, from parent_qualified_name) | +-- imports edges (file → imported symbol, per-language extraction) | +-- invokes edges (function → called function, line-range matching) | +-- edges (source_id, target_id, edge_type) | +-- bm25s.index(repo) → pickle file (file-level BM25) // All stored in one file: config.sqlite_db // Tables: symbols, edges, indexed_files, index_meta, cocoindex_symbols, code_embeddings_vec ———————————————————————————————— // Legacy (indexing_backend="legacy") — mtime-based index_builder.build_index(repo, db) iter_source_files → mtime check → extract_symbols → SQLite pilot/mcp/code_locator_runtime.ensure_index_matches_repo() → force rebuild on HEAD mismatch graph_builder → edges bm25s.index → pickle (optional: VectorSearchClient → cocoindex-code daemon)
Symbol Extractor
Tree-sitter → 9 languages
Extracts classes, functions, methods with qualified names, signatures, parent relationships, and line ranges. Per-language walkers for Python, JS/TS/JSX/TSX, Java, Go, Rust, C#.
SymbolRecord: name, qualified_name, type, file_path, start_line, end_line, signature, parent
Graph Builder
3 edge types, 2nd tree-sitter pass
contains: class → method (from parent_qualified_name).
imports: per-language import extraction (Python, JS/TS, Go, Java).
invokes: call sites matched to definitions by line range.
CocoIndex Pipeline
Dual-target, fingerprint incremental
One pass produces both embeddings (sqlite-vec) and symbols. @coco.fn(memo=True) skips unchanged files. Stable content-hash IDs for row-level reconciliation. Configurable chunk_size/overlap.
Git State Guard
repo_path + branch + head_commit
Runtime code records the git state the index was built from in index_meta. If the repo's current HEAD changes after checkout, merge, or pull, the next query force-refreshes before serving results. This fixes stale-index correctness without adding cross-branch search yet.

03 / Agent Loop

LLM Tool-Calling Loop

The agent loop runs up to 5 LLM turns and 8 tool calls. The LLM sees the planned change + top 20 symbols by connectivity as codebase context. It calls tools iteratively until it has enough evidence, then outputs a JSON answer validated against the allowlist.
run_agent_loop(planned_change, tools, config, db) State: tool_log: list[ToolInvocation] ← structured record of every call allowed_files: set[str] ← grows with each tool result allowed_symbols: set[str] ← grows with each tool result System prompt includes: - PlannedChange intent + business_context - Top 20 symbols by connectivity (naming conventions) - Retry context (if Agent C rejected previous attempt) Loop (max 5 turns, 8 tool calls): | +-- MCP host LLM orchestrates tools directly (no nested LLM) | +-- if tool_calls: | +-- dispatch to validate_symbols / search_code / get_neighbors | +-- update allowlists from results | +-- serialize results (max 4000 chars) into messages | +-- continue loop | +-- if no tool_calls (final answer): | +-- _parse_and_validate_answer | +-- extract JSON → check each entry against allowlists | +-- accepted → FoundComponent + Provenance | +-- rejected → rejected_hallucinations[] | +-- if loop exhausted without answer: +-- forced completion (no tools, text-only response) Output: AgentResult found_components, tool_log, reasoning_trace, rejected_hallucinations, total_latency_ms, total_token_usage
04 / Tool Registry

Three Deterministic Tools

Tool 1 — Vocabulary Bridge
validate_symbols
LLM proposes candidate symbol names → 2-stage rapidfuzz matching against the full symbol index.

Stage 1: fuzz.ratio on name + qualified_name, top 100 survivors (score > 30).
Stage 2: fuzz.WRatio on survivors. Cross-convention matching (camelCase ↔ snake_case via underscore stripping). Threshold: 80.
candidates: list[str] → ValidatedSymbol[] (matched_symbol, match_score, symbol_id)
Tool 2 — Multi-Channel Retrieval
search_code
3-channel search fused with weighted RRF:

Ch 1: BM25 (always, w=1.0) — file-level text search via bm25s
Ch 2: Graph (if symbol_ids, w=1.2) — seed symbol + 1-hop neighbors
Ch 3: Vector (if available, w=0.6) — sqlite-vec KNN cosine similarity

Vector results enriched with symbol names by matching file+line range against the symbol index.
query: str, symbol_ids?: int[] → RetrievalResult[] (file, line, score, method)
Tool 3 — Structural Context
get_neighbors
Bidirectional 1-hop ego-graph query (single SQL round-trip). Sorted by edge priority:

invokes (0) > imports (1) > inherits (2) > contains (3)

Within same type, backward (callers) before forward (callees). Capped at 10 neighbors.
symbol_id: int → NeighborInfo[] (symbol, file, edge_type, direction)
05 / Retrieval Architecture

3-Channel RRF Fusion

// Weighted Reciprocal Rank Fusion (fusion/rrf.py) // Score formula: score[key] += weight / (k + rank + 1) where k=60 query: "Add rate limiting to route registration" | +-- BM25 (w=1.0) # file-level text search, always runs | scaffold.py [rank 0] score += 1.0/(60+1) = 0.0164 | app.py [rank 1] score += 1.0/(60+2) = 0.0161 | +-- Graph (w=1.2) # seed symbols + 1-hop, if symbol_ids given | Scaffold.route [rank 0, score=1.5] += 1.2/(60+1) = 0.0197 | App.add_url_rule [rank 1, score=1.0] += 1.2/(60+2) = 0.0194 | +-- Vector (w=0.6) # sqlite-vec KNN, cosine similarity | scaffold.py:336 [rank 0] += 0.6/(60+1) = 0.0098 | app.py:602 [rank 1] += 0.6/(60+2) = 0.0097 | +-- RRF Merge dedup by (file_path, line_number) scaffold.py: 0.0164 + 0.0197 + 0.0098 = 0.0459 ← method="bm25+graph+vector" app.py: 0.0161 + 0.0194 + 0.0097 = 0.0452
06 / Safety

Hallucination Boundary

Every tool call adds to the allowlists. The final answer is validated entry-by-entry. When search_code finds a file, ALL symbols in that file are auto-populated into the allowlist via a DB query — closing the gap where BM25 discovers a file but its symbols weren't explicitly validated.
Allowlist Growth
Additive across all tool calls
validate_symbols → adds matched_symbol to allowed_symbols
search_code → adds file_path to allowed_files + auto-populates symbols
get_neighbors → adds both files and symbols
Rejection
4 rejection categories
hallucinated_both — neither file nor symbol in allowlist
hallucinated_file — symbol OK, file not seen
hallucinated_symbol — file OK, symbol not seen

Symbol check uses suffix/containment match (handles LLM adding module prefixes).

08 / System Integration

Decision Ledger ↔ Code Locator

The two systems are decoupled by design. The symbol name is the shared join key. The pipeline orchestrator is the only component that talks to both — it reads matched symbols from AgentResult.found_components and writes maps_to edges into the SurrealDB ledger graph. Code Locator never imports or calls SurrealDB. Five contracts enable Jin and Silong to work independently. Ref: decision-ledger-integration.md
// End-to-end data flow (from 2026-03-24 whiteboard) Input (transcript, Notion doc, manual — any "product discussion") | +-- Agent A: Extract PlannedChange + InputSpan | +-- Code Locator (Agent B — runs untouched) | InputSpan → [Vocabulary Bridge] → ValidatedSymbol[] | → BM25 + Graph + Vector(CocoIndex) → RRF Fusion | → FoundComponent[] | symbol: "Flask.wsgi_app" ← join key | file: "src/flask/app.py:1618" | confidence: 0.92 | provenance: {channels: [bm25,graph,vector], bridge_score: 95} | neighbors: [{symbol: "Flask.__call__", edge: "contains"}] | +-- Orchestrator (server.py — only component that talks to both) | | | +-- Contract 1: AgentResult → CodeLocatorPayload (adapter converts) | | CodeRegion { file_path, symbol, type, start_line, end_line, | | purpose, neighbors } | | SpanMapping { span, intent, symbols, code_regions, dep_edges } | | | +-- Contract 3: SymbolDecisionRequest ("have we seen this code before?") | | symbols: [comp.symbol for comp in found_components] | | → SymbolDecisionResponse { decisions, symbols_with_hits } | | | +-- Contract 5: link_commit (idempotent, auto-triggered before queries) | → SyncStatusResponse { synced, regions_updated, reason } | +-- Decision Ledger (SurrealDB embedded, file:// storage) | | Graph: intent --(maps_to)--> symbol --(implements)--> code_region | | intent --(touches)--> code_region (denormalized O(1) reverse) | (implemented_by) | v | commit | + vocab_cache (BM25 on prior intents, speeds repeated lookups) + CHANGEFEED on code_region + intent (staleness detection) + ledger_sync (single-row sync cursor for idempotent link_commit) // Status derived at query time via content-hash comparison (not stored) reflected = code_region.content_hash matches HEAD drifted = content_hash mismatch (code changed since decision) pending = decision exists, no commit linked yet ungrounded = decision has no mapped code regions at all // $live_hashes: caller reads file regions from disk, builds {file: sha256} map
5 Integration Contracts
Three contracts enable independent work. Two more were added for reverse traversal and idempotent sync. All types in pilot/demo2/contracts.py.
Contract 1 — Handoff
CodeLocatorPayload
Code Locator → Decision Ledger ingestion. Adapter converts AgentResult to CodeLocatorPayload with CodeRegion (file_path, symbol, type, start/end_line, purpose, neighbors), SpanMapping (intent, symbols, dependency_edges), plus usage stats and rejected count.
pilot/demo2/adapters/code_locator_adapter.py • agent_result_to_payload()
Contract 2 — Feedback Loop
LedgerSearchRequest / Response
Decision Ledger → Code Locator generation. BM25 search on prior intents → graph walk to code regions. Vocab cache provides fast-path for repeated queries. Returns LedgerSearchHit with intent, symbols, code_regions, confidence, last_seen.
SurrealQL: intent WHERE description @0@ $query → maps_to → symbol → implements → code_region
Contract 3 — Symbol Lookup
SymbolDecisionRequest / Response
Orchestrator takes matched symbols from FoundComponent[] and queries the ledger: "have we seen this code before?" Returns AssociatedDecision with status (reflected/drifted/pending/ungrounded), drift_evidence, and symbols_with/without_hits.
SurrealQL: symbol WHERE name IN $symbols → <-maps_to<-intent + →implements→code_region
Contract 4 — Reverse Traversal
FileDecisionRequest / Response
Given a file path, find all intents that touch symbols in it. Uses denormalized touches edge for O(1) reverse lookup (vs 3-hop walk). Primary query for /detect_drift. Returns FileDecisionHit + unmatched_symbols.
SurrealQL: code_region WHERE file_path = $f → <-touches<-intent (indexed on out)
Contract 5 — Sync Cursor
SyncStatusRequest / Response
Idempotent link_commit. Single-row ledger_sync table tracks last synced commit per repo. Fast-path: if last_synced_commit == commit_hash, returns already_synced. Auto-triggered by /search_decisions and /detect_drift before any read.
UPSERT ledger_sync SET last_synced_commit = $hash WHERE repo = $repo

09 / MCP Server

bicameral-mcp — 4 Tools over stdio

The MCP server bridges the decision ledger and code locator for AI agents. Adapter layer swaps mock/real backends via env vars (USE_REAL_LEDGER, USE_REAL_CODE_LOCATOR). Currently Phase 0 (mocks).
MCP Tool
decision_status
List all decisions with current status (reflected/drifted/pending/ungrounded). Returns code regions, drift evidence, blast radius. Read-only.
filter="all", since=None, ref="HEAD"
MCP Tool
search_decisions
Semantic search over decisions. Auto-triggers link_commit for sync. Returns matches with confidence + suggested reviews for drifted/pending.
query: str, max_results=10, min_confidence=0.5
MCP Tool
detect_drift
Given a file, find all decisions referencing it and check for drift (content-hash mismatch). Lists undocumented symbols. Auto-triggers link_commit.
file_path: str, use_working_tree=True
MCP Tool
link_commit
Heartbeat: ingest a commit into the ledger. Updates code regions, reflects or drifts decisions. The only write path into the ledger.
commit_hash="HEAD"
SHARED MCP TYPE — CodeRegionSummary
file_path: str • symbol: str • lines: tuple[int, int] • purpose: str
Every MCP tool response that references code uses this type. Normalized interface between both backends.
